# x64 architecture invariants

Drafted during Phase 3 Stage 6 (`docs/phase3-review-plan.md`). Seeded from
what each sub-stage actually confirms, extended as 6b/6c land — same
convention `.claude/rules/memory.md` established for Phase 2 Stage 4.

## SYSRET non-canonical-RCX class — closed structurally, not by a runtime check

`architecture/x64/interface.zig` bounds `user_memory_range` one page short
of the canonical/non-canonical address-space boundary (lines ~176-185).
Since `rcx` at syscall entry always equals the user address the `syscall`
instruction executed from, and that address space can no longer reach the
boundary-adjacent page, `rcx` can never be non-canonical when
`syscallEntry`'s assembly (`user/root.zig`) pops it back out before
`sysretq` — this is a structural fix (illegal states unrepresentable), not
a per-return validation check. Confirmed (Stage 6a) that no code in
`user/` (generic syscall dispatch) ever rewrites `SyscallFrame.rip` before
return, closing the one gap the address-space bound alone doesn't cover
(a handler explicitly redirecting the return address). If a future
syscall ever needs to redirect execution on return (e.g. a hypothetical
`exec`-like syscall), it must validate the new RIP against
`user_memory_range` before writing it — the structural mitigation only
protects the unmodified-passthrough case.

## Exception-vector classification: assembly is the source of truth

`asm/interruptHandlers.S`'s `INTERRUPT_HANDLER .HasError/.NoError`
invocations are the actual, hardware-facing classification of which
exception vectors push an error code. `interrupts/Interrupt.zig`'s
`hasErrorCode()` must agree with it exactly (Stage 6a fixed a case where
it didn't, for vectors 21/29/22-28/31) — check the `.S` file first if the
two ever appear to disagree, not the Zig enum's doc comments alone.

## Per-process fault isolation now covers every exception vector — FIXED

Originally found (Stage 6a) as a severe documented-not-fixed gap:
`interrupts/handlers.zig`'s `unhandledException()`'s `.user` branch
`std.debug.panic`d the entire kernel for any exception other than a page
fault.

**The design**: `Interrupt.exceptionDisposition()` (in `interrupts/
Interrupt.zig`) maps every named exception vector to one of two
dispositions:
- **`.isolate: u8`** — terminate only the calling thread with the given
  exit status (`user.Process.ExitStatus`'s `128 + POSIX signal`
  convention, shared with `process_kill` and the page-fault handler — see
  `.claude/rules/user-boundary.md`). The default and safe choice for
  *any* exception raised from user mode: whatever caused it is confined
  to that one process.
- **`.always_fatal`** — reserved for exactly two vectors, `double_fault`
  and `machine_check`: the CPU detected a failure severe enough that it
  couldn't reliably keep running *any* code, including this kernel's own
  isolation machinery. Panics regardless of privilege level.

Every named vector is listed explicitly (not inferred from a numeric
range) so the choice is visible per-vector. Because `Interrupt` is a
non-exhaustive enum (must represent arbitrary raw vector numbers via
`@enumFromInt`), Zig can't make this switch a compile error for a future
named vector the way an exhaustive enum would — so the `else` fallback
itself has to be the safe choice (`.isolate` with `sigsegv`), never
`.always_fatal` and never `unreachable`. This is the actual "future-proof"
property: a vector nobody thought to add to the table explicitly still
gets contained to one process, not a kernel panic.

**Verified end-to-end, not just compiled**: `apps/
itest_illegal_instruction` (test-only fixture) deliberately executes an
architecture-specific illegal-instruction encoding
(`ud2`/`udf #0`/`unimp`); `testing/integration.test.zig`'s new test spawns
it and asserts the exit status is `sigill` (132). Before this fix, this
exact test would have panicked the entire kernel rather than reaching the
assertion — the log line confirming this actually happened in QEMU:
`unhandled user-mode exception invalid_opcode, killing process (exit
status 132)`, followed by the suite completing normally.

**For Stage 9 (`architecture/arm/`)**: replicate this same philosophy
(isolate-by-default, `always_fatal` reserved for CPU-detected
catastrophic failure equivalents only) for arm's own exception vectors —
don't assume arm's fault handling already has equivalent isolation until
checked.

## Interrupt entry/exit register discipline

`_common_interrupt_handler` (`asm/interruptHandlers.S`) pushes/pops all 15
GPRs symmetrically and decides `swapgs` by reading `CS.RPL` at a fixed
stack offset (`24(%rsp)`) both at entry and exit — valid because x86-64
unconditionally pushes `SS:RSP` for every interrupt regardless of
privilege-level change (unlike 32-bit mode), so the offset never varies.
`interruptDispatch` (Zig side) additionally lazy-saves/restores the
interrupted task's extended (SSE/xsave) state around user-mode
interrupts. **Stage 6a's open question, resolved by Stage 6b**:
`scheduling.zig`'s `switchTask`/`switchTaskNoSave`/`call`/`callNoSave` are a
stackful-coroutine-style switch — they save/restore `rsp` and `rbp` (and
an explicit return address via the `lea 1f(%rip)` trick) rather than
mutating an `InterruptFrame` in place. A task switch triggered from inside
`interruptDispatch` therefore doesn't disrupt its own deferred epilogue
(the SSE-state reload): each task's Zig call stack, including any pending
`defer`, is preserved independently and only resumes when that task's
saved stack pointer is switched back to — at which point `Task.Current`
necessarily refers to that same task again. **Still open**: the exact
ordering of `Task.Current`/`PerTask.setCurrentTask` relative to
`switchTask`'s asm (does the GS-base update happen before or after the
stack swap, and is there a window where they're inconsistent) is
scheduler-side code in `task/`, not yet reviewed — confirm in Stage 8.

## `PageTable.unmap()`'s asymmetric level cleanup is deliberate

`top_level_decision` only gates freeing an emptied level-3 table; emptied
level-2/level-1 tables are always freed regardless. Deliberate: level-3
tables for the kernel half of the address space are shared across every
process's page table (`copyTopLevelIntoPageTable` copies level-4 entries,
which point at them, verbatim into every new address space), so kernel-
context callers pass `.keep` to avoid tearing down a structure other
address spaces still reference; user-context callers pass `.free` since a
user address space's level-3 tables are never shared. Documented directly
on `unmap()` now (Stage 6b) — this note exists so the reasoning doesn't
need re-deriving from the caller sites again.

## Debug-print copy-paste bugs are a recurring pattern here — check indices/fields match their own label

Stage 6b found two: `PageFaultErrorCode.print()` checked
`self.instruction_fetch` four times instead of the four distinct fields
each `if` was labeling (`protection_key`/`shadow_stack`/`hlat`/
`software_guard_exception`); `PageTable.print()`'s 2MiB-huge-entry
non-raw-flag branch printed `level3_index`/`level3_range` where the
sibling `print_raw_flag_bits` branch two lines above correctly used
`level2_index`/`level2_range`. Both fixed. Neither was a functional/
security bug (both are debug-display only), but both are the same shape:
copy-pasted print statements with a stale variable name from the
level/field above. Worth a specific grep pass (look for repeated variable
names across adjacent `if`/branch blocks in `print()`/`format()` functions)
in any not-yet-reviewed file with similar multi-level or multi-field print
code.

**Stage 6c checked `info/cpu_id.zig`** (3,967 lines, CPUID feature-bit
display) against this pattern: it has no `print()`/`format()` functions at
all, so the copy-paste risk didn't show up as duplicated print statements —
instead it showed up as copy-pasted doc-comment labels naming the wrong
CPUID leaf/register (`capture80000007H()`'s EDX-block "reserved" comments
said `CPUID.80000001H: ECX[N]` instead of `CPUID.80000007H: EDX[N]`;
`max_hypervisor_leaf`'s doc comment described `CPUID.80000001H: ECX`
instead of the `CPUID.40000000H: EAX` it's actually populated from). Both
fixed, both comment-only with zero functional impact. Same underlying
lesson: when a block of near-identical lines gets written by copying the
previous one and editing in place, check that every label was actually
updated, not just the code.

**Stage 6c also found the same "dead code invisible to the build" bug a
fourth time**: `apic/LAPIC.zig`'s `LVTErrorRegister.vector` field was typed
`x64.InterruptVector`, a type that doesn't exist anywhere in the codebase
(the real type is `x64.interrupts.Interrupt`, used correctly by every other
`vector` field in the same file). Compiled cleanly because
`LVTErrorRegister`/`readLVTErrorRegister`/`writeLVTErrorRegister` are never
called anywhere — Zig only semantically analyzes a function/type body once
it's referenced. Fixed. This is the same class as Stage 5's
`Arena.deinit()` and Stage 6a's `hasErrorCode()` bugs: worth remembering
that `zig build check`'s clean compile says nothing about code no call site
reaches.
